# Пояснительная записка к проекту «Quiz Arena»

**Тема:** веб-платформа для проведения квизов в реальном времени (аналог Kahoot).
**Тип:** учебный проект (MVP — минимально работающий продукт).

Документ написан простым языком: незнакомые термины поясняются сразу же по ходу текста.
В конце есть словарик основных понятий.

---

## 1. Что это за приложение и что оно умеет

**Quiz Arena** — это сайт, на котором один человек (**организатор**) создаёт викторину
и проводит её вживую, а другие люди (**участники**) подключаются со своих телефонов или
компьютеров и отвечают на вопросы. Всё происходит **одновременно и в реальном времени**:
как в телевизионной викторине, где вопрос появляется у всех сразу, а на ответ даётся
ограниченное время.

Как это выглядит по шагам:

1. Организатор регистрируется и создаёт квиз: пишет вопросы, варианты ответов, отмечает
   правильные, задаёт время на каждый вопрос.
2. Организатор запускает квиз. Система выдаёт **код комнаты** (например `KX7P2M`).
3. Участники заходят на сайт, вводят этот код и своё имя — попадают в «комнату ожидания».
4. Организатор нажимает «Начать игру». Первый вопрос появляется **у всех сразу**.
5. Пока идёт таймер, участники выбирают ответ. Кто ответил быстрее и правильно — получает
   больше очков.
6. После каждого вопроса показывается таблица лидеров. В конце — итоговый рейтинг,
   и все результаты сохраняются в базу данных.
7. В личном кабинете остаётся история: у организатора — проведённые игры, у участника —
   в каких играх он участвовал и какое место занял.

## 2. Обоснование выбора технологий

Технология — это готовый инструмент, который берёт на себя часть работы. Мы выбирали такие
инструменты, чтобы (а) точно попасть в требования ТЗ, (б) не усложнять проект и (в) чтобы
всё запускалось без сложной настройки.

| Что нужно | Что выбрали | Почему именно это |
|-----------|-------------|-------------------|
| Основа сайта (страницы + сервер) | **Next.js** (React) | Один проект содержит и внешний вид (то, что видит пользователь), и серверную часть. Это соответствует ТЗ (React/Next.js + Node.js) и требует меньше настройки для новичка. |
| Реальное время | **Socket.IO** | ТЗ требует WebSocket — технологию, которая держит **постоянную связь** между сервером и браузером. Socket.IO — популярная библиотека поверх WebSocket с простым API и готовыми «комнатами». |
| Хранение данных | **SQLite** (через встроенный в Node модуль `node:sqlite`) | База данных в **одном файле**, без установки отдельной программы-сервера БД. Встроенный модуль не нужно собирать из исходников (это была проблема с другими библиотеками на Node 24). |
| Внешний вид (стили) | **Tailwind CSS** | Позволяет быстро и аккуратно оформлять страницы, не заводя множество отдельных файлов стилей. |
| Пароли | **bcryptjs** | Библиотека для **хеширования** паролей (см. словарик). Написана на чистом JavaScript, не требует сборки. |
| Вход в аккаунт (сессии) | Своя простая реализация на cookie | Чтобы не тянуть тяжёлые библиотеки ради учебного проекта. |
| Шрифты | **Inter** и **Space Grotesk** (через `next/font`) | Аккуратная типографика; шрифты скачиваются при сборке и раздаются с нашего сервера. |

**Ключевое техническое решение — «кастомный сервер».** Обычный Next.js-сайт может работать
в режиме, где сервер «просыпается» только на время одного запроса. Но для реального времени
нужен процесс, который **работает постоянно** и держит связь со всеми участниками. Поэтому
мы написали свой файл запуска [`server.js`](server.js), который поднимает Next.js и Socket.IO
**вместе на одном порту**.

## 3. Архитектура — как всё устроено внутри

Приложение состоит из трёх больших частей, которые общаются между собой.

```
   Браузер участника/организатора  ←→  Сервер (Next.js + Socket.IO)  ←→  База данных (SQLite)
        (страницы, кнопки)              (логика игры, проверки)          (файл data/quiz.db)
```

- **Браузер (клиент)** — то, что видит человек: страницы, формы, кнопки. Написан на React.
- **Сервер** — «мозг» приложения: проверяет пароли, хранит логику игры, решает, кто сколько
  очков получил, и рассылает вопросы всем участникам одновременно.
- **База данных** — «долговременная память»: пользователи, квизы, вопросы, результаты игр.

### Важная идея: разделение «постоянного» и «временного»

- **Постоянные данные** (пользователи, квизы, итоги игр) хранятся в **базе** — они не должны
  пропадать. За это отвечают файлы в папке [`lib/`](lib).
- **Временное состояние во время игры** (кто прямо сейчас в комнате, какой идёт вопрос,
  сколько у кого очков в этот момент) хранится **в памяти сервера** — [`lib/game.js`](lib/game.js).
  Это быстро и удобно, а по окончании игры итоги записываются в базу.

### Структура папок

```
проект/
├─ server.js              # запуск: Next.js + Socket.IO на одном порту
├─ app/                   # страницы сайта (Next.js App Router)
│  ├─ page.js             # главная
│  ├─ login, register     # вход и регистрация
│  ├─ dashboard/          # личный кабинет с историей
│  ├─ organizer/          # создание квиза и экран ведущего
│  ├─ join/               # экран «ввести код комнаты»
│  ├─ play/[code]/        # игровой экран участника
│  ├─ game/[id]/          # страница результатов одной игры
│  └─ api/                # серверные обработчики (вход, регистрация, создание квиза)
├─ components/            # переиспользуемые части: иконки, шапка, кнопка выхода
├─ lib/                   # серверная логика
│  ├─ db.js               # подключение к базе и создание таблиц
│  ├─ auth.js             # регистрация, пароли, сессии
│  ├─ quizzes.js          # сохранение и чтение квизов
│  ├─ game.js             # активные игры в памяти
│  └─ history.js          # запросы истории для кабинета
├─ socket/handlers.js     # вся логика игры в реальном времени
└─ data/quiz.db           # файл базы (создаётся автоматически)
```

## 4. База данных — как хранятся данные

Данные разложены по **таблицам** (как листы в Excel). Таблицы связаны между собой ссылками
(например, вопрос «знает», к какому квизу он относится).

| Таблица | Что хранит |
|---------|-----------|
| `users` | Пользователи: почта, зашифрованный пароль, имя, роль (организатор/участник) |
| `sessions` | Активные входы: какой «пропуск» (токен) какому пользователю принадлежит |
| `quizzes` | Квизы: название, категория, время на вопрос, кто создал |
| `questions` | Вопросы квиза: текст, тип (один/несколько ответов), картинка |
| `answers` | Варианты ответов к вопросам и пометка, какой правильный |
| `games` | Запуски игр: код комнаты, статус, когда началась и закончилась |
| `game_results` | Итоги игр: кто сколько набрал и какое занял место |

Все таблицы создаются автоматически при первом запуске — см. [`lib/db.js`](lib/db.js).

## 5. Как работает главная механика — игра в реальном времени

Это самая важная и сложная часть проекта. Она построена на **обмене сообщениями** между
браузером и сервером через Socket.IO. Логика собрана в [`socket/handlers.js`](socket/handlers.js).

Пошагово:

1. **Создание комнаты.** Организатор открывает экран ведущего → браузер отправляет серверу
   сообщение `host_create`. Сервер берёт квиз из базы, придумывает уникальный код комнаты
   и создаёт «игру» в своей памяти.
2. **Подключение участников.** Участник вводит код → сообщение `player_join`. Сервер проверяет,
   что комната существует и игра ещё не началась, и добавляет игрока. Всем в комнате
   рассылается обновлённый список участников.
3. **Показ вопроса.** Организатор жмёт «Начать» / «Следующий вопрос» → сервер отправляет
   **всем одновременно** сообщение `question` с текстом, вариантами и **временем окончания**
   (`deadline`). Важно: правильные ответы в этом сообщении **не передаются**, чтобы их нельзя
   было подсмотреть в браузере.
4. **Приём ответов.** Участник выбирает вариант → сообщение `submit_answer`. Сервер проверяет,
   что **время ещё не вышло** (сравнивает с `deadline`), и начисляет очки. Проверка времени
   делается **на сервере**, потому что данным из браузера доверять нельзя.
5. **Начисление очков.** За правильный ответ даётся от 500 до 1000 очков: чем быстрее ответил,
   тем больше. Это делает игру динамичной.
6. **Показ правильного ответа.** Когда время вышло (или все ответили), сервер рассылает
   `reveal` — подсвечивает правильные варианты и показывает таблицу лидеров.
7. **Завершение.** После последнего вопроса сервер сохраняет итоги в базу (таблицы `games`
   и `game_results`) и рассылает финальный лидерборд.

### Как история привязывается к аккаунту

Когда участник входит в игру, будучи авторизованным, сервер определяет его аккаунт по
**cookie сессии**, которую браузер автоматически присылает вместе с подключением. Благодаря
этому сыгранная партия попадает в личную историю участника. Гость без входа просто не
привязывается к аккаунту. Такой способ надёжнее, чем передавать номер пользователя из
браузера, потому что данные из браузера можно подделать.

## 6. Безопасность и надёжность

Даже в учебном проекте важно показать аккуратное отношение к данным:

- **Пароли не хранятся в открытом виде.** Они проходят через **хеширование** (bcrypt):
  в базе лежит не сам пароль, а его необратимый «отпечаток».
- **Проверка времени — на сервере.** Ответ, присланный после окончания таймера, не
  засчитывается, даже если участник схитрит в браузере.
- **Правильные ответы не отправляются в браузер** до момента показа результата.
- **Доступ к чужим данным закрыт.** Организатор не может открыть чужой квиз или результаты
  чужой игры — сервер проверяет, что данные принадлежат именно ему.
- **Секреты не хранятся в репозитории.** Файл базы данных и папка `node_modules` исключены
  из git через `.gitignore`.

## 7. Этапы работы над проектом

1. **Настройка окружения.** Установка Node.js и git, создание каркаса Next.js, первый запуск.
2. **База данных.** Проектирование таблиц и их создание в `lib/db.js`.
3. **Авторизация.** Регистрация, вход, роли, сессии, личный кабинет.
4. **Конструктор квизов.** Форма создания: вопросы всех типов, картинки, время на вопрос.
5. **Игра в реальном времени.** Комнаты по коду, показ вопросов по таймеру, приём ответов,
   очки за скорость, лидерборд, сохранение итогов — ядро проекта.
6. **История в кабинете.** Списки проведённых игр и участия, страница результатов игры.
7. **Дизайн.** Единый тёмный «технологичный» стиль на синей айдентике, замена всех эмодзи
   на аккуратные иконки, адаптивная вёрстка.
8. **Публикация.** Оформление репозитория на GitHub, README, пояснительная записка.

На каждом этапе приложение проверялось на запуск без ошибок; ключевые сценарии (регистрация,
создание квиза, полная партия на нескольких игроков) тестировались отдельно.

## Словарик терминов

- **Клиент / браузер** — программа, в которой пользователь открывает сайт (Chrome и т.п.).
- **Сервер** — программа, которая обрабатывает запросы и хранит логику; работает «за кадром».
- **База данных (БД)** — организованное хранилище данных. У нас — файл SQLite.
- **WebSocket** — способ держать **постоянную двустороннюю связь** между браузером и сервером,
  чтобы мгновенно обмениваться сообщениями (нужно для реального времени).
- **Socket.IO** — библиотека, которая упрощает работу с WebSocket и даёт «комнаты».
- **Хеширование** — превращение пароля в необратимый «отпечаток», чтобы не хранить его открыто.
- **Сессия / токен** — «электронный пропуск», который подтверждает, что пользователь вошёл.
- **Cookie** — небольшой кусочек данных, который браузер хранит и присылает серверу
  (мы храним в нём пропуск сессии).
- **Роль** — тип пользователя: организатор (ведёт игру) или участник (играет).
- **Лидерборд** — таблица участников, отсортированная по очкам.
- **MVP** — минимально работающая версия продукта со всеми ключевыми функциями.
```
